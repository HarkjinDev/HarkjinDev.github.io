3. 객체지향 설계 5대 원칙 (SOLID)

  • 정의: 유지보수성과 확장성을 높이는 객체지향 설계의 5가지 핵심 원칙
  • 키워드: SRP, OCP, LSP, ISP, DIP, 클린 코드
  • 분류: 프로그래밍
  • SOLID 5대 원칙
    원칙핵심 내용기법
    SRP(단일 책임)클래스는 하나의 변경 이유만 가져야 함클래스 분리·모듈화
    OCP(개방-폐쇄)확장에는 열려있고 수정에는 닫혀있어야 함추상화·인터페이스·상속
    LSP(리스코프 치환)자식 클래스는 부모 클래스를 대체할 수 있어야 함계층 구조 재설계·계약 준수
    ISP(인터페이스 분리)클라이언트는 사용하지 않는 인터페이스에 의존 금지인터페이스 세분화·역할 분리
    DIP(의존성 역전)고수준 모듈이 저수준 모듈에 의존하면 안 됨추상화·의존성 주입(DI)
  • SOLID 예시(인터넷 쇼핑몰):
  • S — 단일 책임 원칙 (SRP)
    항목내용핵심
    원칙하나의 모듈은 하나의 역할만역할 분리
    주문 관리주문·취소만 담당단일 책임
    결제 처리카드·간편결제만 담당단일 책임
    배송 관리택배·배송추적만 담당단일 책임
    회원 관리가입·인증만 담당단일 책임
    핵심 포인트결제 로직 수정 시 주문·배송·회원 코드에 영향 없음독립 수정 가능
  • O — 개방-폐쇄 원칙 (OCP)
    항목내용핵심
    원칙기존 코드 수정 없이 확장확장 개방·수정 폐쇄
    Payment (Closed)결제 인터페이스 자체는 수정 불가인터페이스 불변
    카드결제·카카오페이 (Closed)기존 구현체 코드 수정 불가기존 코드 불변
    애플페이 (Open)Payment 구현 새 클래스만 추가신규 추가만 허용
    핵심 포인트애플페이 추가 시 기존 코드 일절 수정 불필요기존 코드 영향 없음
  • L — 리스코프 치환 원칙 (LSP)
    항목내용핵심
    원칙자식은 부모를 완전히 대체 가능대체 가능성
    Delivery (부모)배송 인터페이스 정의공통 계약
    일반배송Delivery 자리에 완전 대체 가능계약 준수
    로켓배송Delivery 자리에 완전 대체 가능계약 준수
    새벽배송Delivery 자리에 완전 대체 가능계약 준수
    핵심 포인트어떤 배송 방식이든 Delivery 타입 자리에 넣어도 정상 동작부모 계약 위반 없음
  • I — 인터페이스 분리 원칙 (ISP)
    항목내용핵심
    원칙필요한 인터페이스만 구현최소 인터페이스
    판매자 → SellerInterface상품등록·재고관리 인터페이스만 구현역할별 분리
    구매자 → BuyerInterface주문·리뷰 인터페이스만 구현역할별 분리
    위반 시 문제구매자가 재고관리·정산 메서드 억지 구현불필요 의존 발생
    핵심 포인트역할별 인터페이스 분리로 불필요한 메서드 구현 강제 없음불필요 메서드 제거
  • D — 의존성 역전 원칙 (DIP)
    항목내용핵심
    원칙구현이 아닌 추상화에 의존추상화 의존
    쇼핑몰 (고수준)DeliveryService 인터페이스에만 의존추상화 의존
    DeliveryService배송 인터페이스 (추상화 계층)의존 역전 지점
    CJ대한통운·한진택배·쿠팡로지스틱스DeliveryService 구현체 (저수준)구현체 교체 자유
    위반 시 문제CJ대한통운 직접 의존 시 택배사 변경에 전체 코드 수정 필요강결합 문제
    핵심 포인트택배사 교체해도 쇼핑몰 핵심 코드는 수정 불필요결합도 감소
  • 인사이트: SOLID는 단순 규칙이 아닌 상호 보완적 원칙으로, OCP는 추상화·다형성으로, DIP는 인터페이스·IoC 컨테이너로 실현되며, 설계 패턴의 이론적 기반.
