3. 마이크로 서비스 아키텍처 (MSA, Micro service Architecture)

  • 정의: 애플리케이션을 독립적으로 배포·확장 가능한 소규모 서비스들의 집합으로 구성하는 아키텍처 스타일
  • 키워드: 독립 배포, 단일 책임, API 통신, 분산 시스템, 서비스 경계, 도메인 주도 설계(DDD)
  • 분류: 소프트웨어 아키텍처
  • 개념도:
  • MSA 설계 원칙:
    원칙설명
    단일 책임각 서비스는 하나의 비즈니스 기능만 담당
    독립 배포다른 서비스에 영향 없이 독립적 배포 가능
    분산 데이터 관리서비스별 독립 DB (Database per Service)
    장애 격리서킷 브레이커로 장애 전파 방지
    API 통신REST·gRPC·메시지 큐로 서비스 간 통신
  • MSA 핵심 패턴:
    패턴설명
    Polyglot Persistence서비스별 적합한 데이터베이스 설계 방식
    API Gateway단일 진입점, 인증·라우팅·집계
    서비스 디스커버리동적으로 서비스 위치(IP:Port) 탐색
    서킷 브레이커장애 서비스 호출 차단, 빠른 실패
    SAGA분산 트랜잭션 일관성 보장 (코레오그래피·오케스트레이션)
    CQRS읽기(Query)·쓰기(Command) 모델 분리
    Strangler Fig모놀리스를 점진적으로 MSA로 전환
  • MSA 도입 시 고려 비용:
    • 네트워크 레이턴시 증가
    • 분산 트랜잭션 복잡성 (SAGA 패턴)
    • 운영 복잡도 (컨테이너·오케스트레이션)
    • 분산 추적·모니터링 필요
  • 인사이트: MSA는 “조직이 아키텍처를 닮는다”는 Conway의 법칙에 따라 팀 구조와 서비스 경계가 일치해야 효과적이며, 작은 팀에서 MSA를 도입하면 오히려 복잡성이 증가.
