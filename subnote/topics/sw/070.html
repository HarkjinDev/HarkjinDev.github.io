2. 모놀리스 아키텍처 (Monolith Architecture)

  • 정의: 애플리케이션의 모든 기능(UI·비즈니스 로직·데이터 접근)이 단일 배포 단위로 통합된 전통적 소프트웨어 아키텍처
  • 키워드: 단일 배포, 단단한 결합, 단순성, 스케일링 한계, 기술 스택 고정, 빅볼오브머드
  • 분류: 소프트웨어 아키텍처
  • 모놀리스 유형:
    유형설명
    모듈러 모놀리스 (Modular Monolith)내부 모듈화된 단일 배포, MSA 전환 중간 단계
    빅볼오브머드 (Big Ball of Mud)구조 없이 뒤얽힌 코드베이스
  • 모놀리스 vs SOA vs MSA 비교:
    구분모놀리식 (Monolith)SOA (Service Oriented)MSA (Microservice)
    개념모든 기능이 하나의 패키지로 구성된 일체형 구조서비스 단위로 구축하고 ESB로 통합하는 전사적 구조독립적인 작은 서비스들의 집합체로 구성된 분산 구조
    통신Method Call (내부 호출)ESB (Enterprise Service Bus), SOAP/WSDLRest API / gRPC, Lightweight Messaging
    DB통합 DB 공유 DB (주로 통합 관리)서비스별 개별 DB
    재사용성코드 수준의 재사용서비스/비즈니스 수준의 재사용컴포넌트 수준의 재사용
    배포/ 확장성전체 재배포 필요 / 수직 확장(Scale-up)서비스 단위 배포 가능 / 중앙 집중형 확장개별 서비스 독립 배포 / 수평 확장(Scale-out)
    보안 관점단일 접점 보안 (방화벽 위주)ESB 중심의 보안 정책 통합 관리서비스별 보안(Zero Trust), API Gateway 보안
    장점개발/배포 단순, 로컬 성능 우수표준 기반 통합, 비즈니스 유연성높은 유연성, 기술 스택 자유도, 장애 격리
    단점부분 장애가 전체 확산, 수정 시 전체 배포중앙 ESB 병목 및 복잡도 증가분산 트랜잭션 관리 어려움, 인프라 복잡성
  • 인사이트: 모놀리스가 항상 나쁜 아키텍처는 아니며, “모듈러 모놀리스”는 MSA의 복잡성 없이 모듈화 이점을 누리는 현실적 대안으로, 스타트업 초기에는 모놀리스로 시작 후 필요 시 MSA 전환이 권장 전략.
