1. 소프트웨어 아키텍처

  • 정의: 프로그램/시스템 컴포넌트, 컴포넌트 간의 상호관계 구조이며 이들을 설계하기 위한 지침과 원리
  • 키워드: IEEE 42010, AD, Stakeholder, Concerns, Viewpoint, View
  • 분류: 소프트웨어 아키텍처
  • 구성요소:
    요소내용
    Architecture Description (AD)* 아키텍처를 기록하기 위한 산출물들의 집합 * 하나의 AD는 시스템에 대한 관심사(Concern)를 나타내며, 이에 대응하는 하나 이상의 이해관계자(Stakeholder)와 연관됨
    이해관계자 (Stakeholder)* 소프트웨어 시스템 개발에 관련된 모든 사람과 조직을 의미 * 고객, 최종 사용자, 개발자, PM, 유지보수자, 마케팅 담당자 등을 포함
    관심사 (Concerns)* 동일한 시스템에 대해 각 이해관계자들이 가지는 서로 다른 의견과 목표 * (예) 사용자: 기능/보안, 유지보수자: 용이성, 개발자: 비용/인력 효율성 등
    관점 (Viewpoint)* 서로 다른 역할이나 책임으로 시스템이나 산출물들을 바라보는 이해관계자들의 견해 * View를 구성하기 위한 규칙을 정의하는 패턴이며, 각각의 View와 1:1로 대응됨
    뷰 (View)* 이해관계자들과 이들이 가지는 생각이나 견해로부터 전체 시스템을 표현한 결과물
  • 소프트웨어 아키텍처 프로세스: <요참모프배>
    단계세부 활동 및 절차
    아키텍처 요구 파악이해당사자 식별 → 품질 속성 식별(기능성, 사용성 등) → 품질 속성 시나리오 작성 → 아키텍처 요구사항 파악 → 아키텍처 리스크 식별 → 아키텍처 자료 검토
    참조 아키텍처 준비업계 모델 조사/검토 → 표준 모델 조사/검토 → 참조 아키텍처 준비 (TRM, SP 등)
    아키텍처 모델링뷰 정의 → 패턴/스타일 설정 → 프레임워크 선정 → 설계 이슈 선정 → 아키텍처 모델링 → 아키텍처 평가
    아키텍처 프로토타이핑유즈케이스 단위 설계 → 컴포넌트 단위 설계 → 기술 이슈 설계 → 구현 → 아키텍처 프로토타이핑 → 프로토타입 팀 구성 → 프로토타입 검토
    아키텍처 배포아키텍처 모델 구조화 → 프로토타입 모델 구조화 → 아키텍처 문서화 → 아키텍처 발표 자료 작성 → 아키텍처 배포 → 아키텍처 문서 검토
