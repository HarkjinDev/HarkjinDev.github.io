4. 데이터베이스 회복기법 (Database Recovery)

  • 정의: 데이터베이스 운영 중 예기치 못한 장애(Failure)가 발생한 경우, 데이터베이스를 장애 발생 이전의 일관된 상태(Consistent State)로 복원시키는 기법
  • 키워드: REDO, UNDO, 로그기반 회복, CheckPoint, 그림자 페이지, 즉시갱신, 지연갱신
  • 분류: 트랜잭션/병행제어
  • 장애유형:
    구분장애 유형상세 설명
    H/WMedia 장애CPU, Memory, Disk 문제 발생되어 데이터가 파손되는 장애
    H/W 성능 저하낮은 사양의 H/W로 인한 Data 처리 속도 지연, 누락
    S/WSyntax 장애프로그램 오류나 사용 용량, 여유 공간 부족으로 발생
    Instance 장애시스템이 비정상적인 요인으로 DB 프로세스 중단
    사용자Fat-finger Error사용자 실수로 테이블 삭제, 잘못된 트랜잭션 처리
    Parameter ErrorDB를 구성하는 주요 Parameter 설정 오류
  • 회복연산:
    • 덤프(dump) : 데이터베이스 전체를 다른 저장 장치에 주기적으로 복사하는 방법
    • 로그(log) : 변경값을 별도의 파일에 기록하는 방법
    • 재실행(Redo) : 오류와 연관된 트랜잭션을 재실행하여 데이터베이스 상태를 복원하는 기법
    • 취소(Undo) : 트랜잭션 장애에 대하여 모든 변경을 취소하여 데이터베이스 상태를 복원하는 기법
  • REDO / UNDO:
    구분정의특징
    REDO장애 발생 시, 진행된 트랜잭션에 대해 로그 기록을 이용하여 재실행하여 복원하는 기법데이터에 대해 일어나는 모든 변화를 기록한 REDO Log 이용. 버퍼와 디스크 사이에 Log 파일 배치
    UNDO장애 발생 시, 모든 변경 사항·Commit 안된 트랜잭션 작업을 취소하여 일관성 보장 기법DB 갱신 내용을 UNDO 세그먼트에 저장하여 UNDO 수행 시 이용. DB 갱신 전 데이터의 상태에 대한 과거 정보 필요
  • 회복기법 유형: <즉언지리/첵그아>
    방법설명
    로그기반 회복기법 (즉시갱신)트랜잭션 수행 중 갱신 결과를 DB와 로그에 즉시 반영. • 장애회복: 로그기반 Undo 수행
    로그기반 회복기법 (지연갱신)트랜잭션 수행 중 로그만 기록, 트랜잭션 종료 후 commit 시점에 DB 반영. • 장애회복: 트랙잭션 종료 : REDO만 실행, 트랜잭션 미종료 : log 무시
    Check Point 기법트랜잭션 변경 로그기록 → 일정시점(체크포인트)마다 트랜잭션 상태 체크, 완료된 트랜잭션 DB에 반영 → 검사점 활용하여 장애 시 신속 복구 • 장애회복: 미완료 트랜잭션: Undo, 완료: Redo 실행
    그림자 페이지 기법트랜잭션이 실행되는 동안 2개의 페이지 테이블 유지·관리. • 트랜잭션 실행 중: 데이터 접근은 current page table만 사용. • commit: current page table 내용 → shadow page table에 저장 • 장애회복: crash 발생 시 current page table 폐기
    ARIES 기법WAL(Write Ahead Logging) 및 LSN(Log Sequence Number)을 이용하는 회복 기법. • 구성 요소: LSN(로그 순차 번호), 오손 페이지 테이블(Dirty Page Table), 체크포인트 레코드 • 장애회복 Analysis(분석) → Redo(재실행) → Undo(취소) 3단계 수행
  • 인사이트: 회복기법 선택 시 시스템 특성에 따라 다르며, 로그기반은 세밀한 복구가 가능하지만 로그 조사 오버헤드가 있고, CheckPoint는 복구 범위를 줄여 성능을 높이며, 그림자 페이지는 로그 없이 페이지 단위 복구가 가능하나 저장 공간 비용이 크다는 트레이드오프를 답안에 반드시 언급 권장
